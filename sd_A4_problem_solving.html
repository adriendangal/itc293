Problem Solving

Applying analytical thinking to technical and business challenges.

Why Problem Solving Matters at NexaCore

Technology work is fundamentally about solving problems. Whether it is debugging code, diagnosing a network issue, or figuring out why a client's system is underperforming, NexaCore employees need to think clearly, break problems into parts, and find workable solutions efficiently.

I have developed structured problem-solving habits through programming study, building a real business, and troubleshooting technical and logistical challenges in my day-to-day study and work.

Examples of Problem Solving in Practice

Programming — Python Parking Records Analyser

For ITC106, I built a command-line Python program called the Utopia Parking Records Analyser. The program read structured data files, processed records, and produced summaries for parking administrators. I had to design the logic for reading and validating data, handle edge cases such as missing or malformed records, and document 15 test cases covering a range of inputs and expected outputs. This project required breaking a large requirement into smaller functions and testing each one systematically.

Business Problem — AI Receptionist Go-to-Market

When I had a working AI receptionist product but no clients, the problem I faced was: how do I reach decision-makers who do not know this product exists? I researched target industries, identified the highest-value segments (dental, real estate), designed a tiered outreach strategy, and iterated on my pitch based on the responses I received. This is a classic problem-solving loop: hypothesise, test, observe, adjust.

Web Development Debugging

Building this portfolio involved troubleshooting CSS layout issues, fixing inconsistent nav behaviour across pages, and ensuring the print stylesheet did not break the screen layout. Each bug was solved by isolating the cause — using browser developer tools, reading the CSS cascade, and testing changes incrementally.

My Problem-Solving Approach

  1. Define the problem clearly — what is the actual issue, not just the symptom?
  2. Break it into smaller parts — which component is failing?
  3. Research and gather information — documentation, peers, or testing.
  4. Try a solution and test it — do not change multiple things at once.
  5. Review and document — record what worked and why.
